# Roadmap：从通用底座到工业级多域平台

本文描述 **Raw Agent SDK / ppeng-agent-core** 的长期产品与技术路线：先夯实「可编排、可观测、可隔离、可扩展」的基础能力，再按领域叠加 **Harness（领域运行时封装）**，最终演进到 **分布式、无单点** 的智能体网络与知识炼油体系。

> 路线图是方向性文件，具体里程碑以 issue / ADR 为准；与实现细节对照见 [`ARCHITECTURE.md`](ARCHITECTURE.md)。

---

## 1. 愿景（North Star）

- **一个工业级超级平台**：同一套会话、工具、审批、沙箱、追踪与领域扩展点，承载多行业工作负载。
- **领域垂直化，而非分叉**：各领域以 **Domain / Harness 包** 形式挂载（工具、Persona、策略、合规钩子），与核心 **API 契约兼容**，避免「每个行业一个 fork」。
- **统一信息收集与提炼**：子节点（会话、任务、外部系统、传感器式集成）产生遥测与产物，汇聚为可检索、可审计、可蒸馏的 **炼油层**，沉淀 **中心知识库**（逻辑上统一，物理上可分区）。
- **天网级电脑矩阵（去中心化）**：多节点自主协作、无主控单点；通过 **联盟式协调**（身份、策略、同步、冲突解决）而非中心化大脑。

---

## 2. 阶段总览

| 阶段 | 名称 | 核心产出 |
|------|------|----------|
| **P0** | 底座硬化 | 稳定 API、安全默认、可观测、多模型、领域扩展点成熟 |
| **P1** | 领域 Harness 1.x | 优先 3～5 个高价值垂直包 + 统一 Harness SDK |
| **P2** | 炼油与知识库 | 跨会话/跨节点的采集、摘要、索引、图谱与保留策略 |
| **P3** | 联邦与无中心拓扑 | 节点发现、任务分片、策略下发、离线/分区容错 |
| **P4** | 工业规模化 | SLA、配额、多租户、合规审计、成本与容量治理 |

---

## 3. P0：基础能力（当前仓库主轴）

与 [`ARCHITECTURE.md`](ARCHITECTURE.md) 对齐，底座应持续强化：

- **运行时与会话**：可靠的多轮编排、流式、人机审批、错误恢复。
- **工具与沙箱**：`sanitizeSpawnEnv`、OS/容器沙箱路径统一；高危操作默认可审计。
- **模型层**：开放兼容适配、路由（文本/VL 等）、成本与降级策略。
- **可观测性**：Trace / 日志 / 指标，可对接外部 OTLP；支持回放与排障。
- **扩展点**：`RuntimeOptions.{extraAgents,extraTools,extraSkills}`、`DomainBundle`、`mergeDomainBundles`、daemon 领域加载器——**Harness 必须只依赖这些契约**。
- **互操作**：MCP、外部 CLI 工具网关、A2UI 等「人机/系统」界面保持版本化协议。

**退出标准（P0）**：在单仓内可重复构建、测试与发布；领域包可独立版本化接入且不修改 core 分叉。

## 4. P1：垂直领域与 Harness 兼容层

### 4.1 领域矩阵（规划）

以下领域按 **Harness 包** 落地（名称可映射为 `domainId` + 工具集 + 策略）；列表去重、合并相近项：

| 方向 | 说明 |
|------|------|
| 医疗 / 金融 | 强合规、审计链、数据最小化；工具默认只读或审批 |
| 模型训练 / MLOps | 实验追踪、artifact、评测集与复现 |
| 运维 / SRE / 集群管理 | 只读优先、变更窗口、回滚剧本 |
| 编码 / 测试 / 安全攻防 | 工作区隔离、密钥与供应链扫描策略 |
| 教育 / 游戏开发 / 游戏自动化 | 内容安全、长会话、资产管线 |
| 办公自动化 | 日历、文档、邮件、工作流；低权限 OAuth |
| 工业自动化 | 时序数据、OPC/PLC 类集成（抽象为适配器） |
| 信息收集 / 舆情分析 | 来源标注、时效、偏见与版权声明 |
| 数据库管理 | 变更 DDL 强审批、影子库、慢查询只读 |
| 组织管理 | RBAC、工单、与身份源同步 |

### 4.2 Harness 定义（目标形态）

- **清单**：`manifest`（工具列表、默认 Persona、合规标签、资源上限）。
- **策略钩子**：领域级审批规则、数据出境检查、日志脱敏。
- **测试**：每个 Harness 自带契约测试（工具 schema、沙箱命令白名单）。

**兼容原则**：Harness 仅通过 **公开扩展 API** 接入；禁止依赖 core 内部模块路径。

## 5. P2：超级炼油中心与中心知识库

- **采集面**：会话摘要、工具 I/O 摘要（脱敏）、trace 事件、外部 ticket/文档链接、指标。
- **提炼**：嵌入、结构化抽取、主题聚类、跨会话 **实体对齐**。
- **存储**：分区与保留策略（合规域独立）；支持「可遗忘」与导出。
- **消费**：控制台/CLI 统一检索；向运行时注入 **经许可的记忆片段**（非隐式全量记忆）。

**退出标准（P2）**：任一子节点产物可在授权范围内被聚合查询，且血缘可追溯。

---

## 6. P3：无中心「电脑矩阵」

- **拓扑**：对等节点 + 可选 **协调平面**（可为多活小服务，但非业务大脑）；任务与状态 **CRDT/事件日志** 或明确冲突策略。
- **安全**：节点身份、mTLS、策略包签名与吊销。
- **韧性**：分区容忍、异步最终一致、人工接管通道。

**退出标准（P3）**：单节点宕机不导致全局瘫痪；禁止出现「只有一个主控进程才能工作」的硬依赖。

---

## 7. P4：工业级运营

- 多租户隔离、配额、计费维度（token、工具调用、存储）。
- SLO、告警、容量规划；灾备与跨区域只读副本。
- 领域合规包（医疗、金融等）的 **认证清单** 与发布闸门。

---

## 8. 非目标与约束（现阶段）

- 不承诺具体日期；大阶段之间可并行（例如 P1 某领域与 P2 存储原型并行）。
- **安全与法律**：舆情、攻防、医疗/金融等方向以 **本地策略与监管要求** 为准，工具默认保守。
- 「天网」为架构隐喻：**不隐含单点失控或绕过用户审批**；人类在环与组织策略始终高于自动化。

---

## 9. 如何参与

- **底座**：issue 标签 `core`、`sandbox`、`observability`。
- **新领域**：先写 Harness 清单与威胁模型，再提 `domain` PR。
- **联邦 / 炼油**：建议先以 ADR 形式讨论数据模型与隐私边界，再动代码。
