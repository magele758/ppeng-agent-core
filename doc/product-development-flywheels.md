# 产品研发飞轮（记录稿）

本文记录一种**多回路**的产品研发视角：信息 → 设计 → 实现 → 数据与部署 → 知识/观测，再通过不同反馈边回到上游。可与本仓库的 Evolution、Web 控制台、Daemon、Self-heal、领域 Agent 等能力对照。

**结构**：**主四环（A–D）**为原先四条业务飞轮；**辅四环（E–H）**为安全合规、成本容量、契约集成、人机质量——与主环常并行、应同样「工件化」。

本仓库的 Evolution 方向审查、能力地图、Agent 调度器模式与上线回归卡点，见 `doc/evolution-flywheel-review.md`。

---

## Loop A — 知识反馈驱动体验与实现

**叙事**：搜集信息 → UI/UX → 功能开发 → 数据存储 → 部署 → **知识沉淀**；知识再作为反馈，更新搜集范围与设计，驱动用户循环迭代。

```
  ┌─────────────┐
  │ 搜集信息     │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ UI / UX     │
  └──────┬──────┘
         ▼
  ┌─────────────┐     ┌─────────────┐
  │ Coding       │────▶│ Data storage│
  └──────┬──────┘     └──────┬──────┘
         │                    │
         ▼                    ▼
  ┌─────────────┐     ┌─────────────┐
  │ Deployment  │────▶│ Knowledge   │
  └─────────────┘     └──────┬──────┘
                              │
         ┌────────────────────┘
         ▼
  （反馈）更新搜集信息 / Design / 用户路径
```

**要点**：强调「知识」对**信息源与设计的再校准**，而不只是文档归档。

---

## Loop B — 数据观测驱动断链修补

**叙事**：通过数据观测发现**产品能力断链**或体验缺口 → 更新信息与设计假设 → 重新设计 → 改代码 → 再进入观测闭环。

```
  ┌─────────────┐
  │ 数据观测     │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 识别断链/缺口│
  └──────┬──────┘
         ▼
  ┌─────────────┐     ┌─────────────┐
  │ 更新信息假设 │────▶│ 更新设计     │
  └─────────────┘     └──────┬──────┘
                              ▼
                       ┌─────────────┐
                       │ 更新代码     │
                       └──────┬──────┘
                              ▼
                       （回到）数据观测
```

**要点**：与 Loop A 的差异在于**入口是度量与漏斗**，而非内容 RSS。

---

## Loop C — 部署侧 SRE / 可观测 → 修复 → 发布

**叙事**：部署环境里的监控、告警、日志 → 定位问题 → 反馈到开发 → 修复 → 再发布。

```
  ┌─────────────┐
  │ Deployment  │
  │ SRE / 可观测 │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 问题发现     │
  └──────┬──────┘
         ▼
  ┌─────────────┐     ┌─────────────┐
  │ Coding 修复  │────▶│ 发布        │
  └─────────────┘     └──────┬──────┘
                              │
                              └──▶（回到）监控面
```

**要点**：偏**可靠性闭环**（MTTR、变更风险、回滚策略）。

---

## Loop D — 主源头定期输入 → 能力维度决策 → 新特性

**叙事**：定期从**主信息源头**拉取输入 → 对照当前**产品能力维度**判断是「提升现有」还是「扩展新域」→ 进入新特性开发，再汇入其它回路。

```
  ┌─────────────┐
  │ 主源头定期   │
  │ 收集信息     │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 能力维度评估 │
  │ 提升 vs 扩展 │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 新特性开发   │
  └──────┬──────┘
         │
         └──▶ 并入 Loop A/B/C 的下游（设计/实现/部署/观测）
```

**要点**：偏**战略节奏**（季度/月度的输入闸门与路线图对齐）。

---

## Loop E — 安全与合规（威胁 → 控制面 → 实现 → 验证 → 反馈）

**叙事**：威胁建模与合规要求 → 安全/隐私设计（控制面、数据流）→ 实现与审计日志 → 渗透、策略扫描或合规检查 → 发现问题则回到设计与代码。

```
  ┌─────────────┐
  │ 威胁 / 合规  │
  │ 要求输入     │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 安全设计     │
  │（控制面/数据）│
  └──────┬──────┘
         ▼
  ┌─────────────┐     ┌─────────────┐
  │ 实现 + 审计  │────▶│ 验证        │
  │（日志/最小权限）│     │渗透/策略扫描 │
  └─────────────┘     └──────┬──────┘
                              │
         ┌────────────────────┘（发现问题）
         ▼
  （反馈）设计修订 / 代码修复 / 发布闸门收紧
```

**要点**：与 Loop C 可并行；强调**左移**（设计阶段定边界）与**可证明**（审计轨迹）。

**本仓库**：`sanitizeSpawnEnv` / `SandboxManager`、`RAW_AGENT_APPROVAL_*`、网关鉴权、Evolution worktree 白名单 `.env` 等；可加强「安全需求 → CI 门禁」清单化。

---

## Loop F — 成本与容量（用量 → 架构策略 → 实现 → 单位经济）

**叙事**：用量、账单、延迟与资源水位 → 驱动架构/模型/缓存/并发策略 → 实现变更 → 再观测单位经济（每请求成本、每用户 Token、每实验 CI 分钟等）。

```
  ┌─────────────┐
  │ 用量 / 账单  │
  │ 延迟 / 水位  │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 容量与成本   │
  │ 策略决策     │
  └──────┬──────┘
         ▼
  ┌─────────────┐     ┌─────────────┐
  │ 实现调优     │────▶│ 再观测       │
  │（缓存/批/限流）│     │单位经济指标 │
  └─────────────┘     └──────┬──────┘
                              │
                              └──▶（回到）用量与预算
```

**要点**：把「能跑」升级为「**划算且可持续**」；与 Loop B 共享指标基建。

**本仓库**：模型调用与 Evolution 并发（`EVOLUTION_CONCURRENCY*`）、会话压缩阈值、外部 AI 工具开关；可显式记录「单次 run-day 预算上限」与 dashboard。

---

## Loop G — 契约与集成（API/MCP/插件 → 版本策略 → 实现 → 契约测试 → 伙伴/CI）

**叙事**：对外 HTTP API、MCP、Cursor 插件等契约 → 版本与兼容策略 → 实现与契约/快照测试 → 集成方或 CI 失败反馈 → 修订契约与实现。

```
  ┌─────────────┐
  │ 契约定义     │
  │ API/MCP/插件 │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 版本与兼容   │
  │ 策略         │
  └──────┬──────┘
         ▼
  ┌─────────────┐     ┌─────────────┐
  │ 实现         │────▶│ 契约测试     │
  └─────────────┘     └──────┬──────┘
                              ▼
                       ┌─────────────┐
                       │ 伙伴 / CI   │
                       │ 集成反馈     │
                       └──────┬──────┘
                              │
                              └──▶（回到）契约与实现
```

**要点**：防「静默破坏」；与 Loop D 新特性衔接时优先更新**契约与迁移说明**。

**本仓库**：`/api/*` 与 stream chunk 类型、A2UI catalog、MCP 工具 schema；PR 前可固定跑契约/回归子集并与 `doc/ARCHITECTURE.md` 对齐。

---

## Loop H — 人机协同质量（Agent 输出 → 抽检/eval → 策略 → 再会话）

**叙事**：模型与 Agent 的产出 → 人类抽检或离线 eval 集打分 → 调整提示词、工具白名单、审批策略 → 再跑真实会话与任务 → 持续迭代。

```
  ┌─────────────┐
  │ Agent 输出   │
  │（会话/工具） │
  └──────┬──────┘
         ▼
  ┌─────────────┐
  │ 抽检 / eval │
  │（人审或自动）│
  └──────┬──────┘
         ▼
  ┌─────────────┐     ┌─────────────┐
  │ 策略更新     │────▶│ 再跑会话     │
  │提示/工具/审批│     │与回归任务    │
  └─────────────┘     └──────┬──────┘
                              │
                              └──▶（回到）Agent 输出质量
```

**要点**：与 Loop A 的「知识」不同，这里强调**行为质量与可重复评测**。

**本仓库**：`doc/evolution` 结果、会话 trace、Skills 路由；可建立小型 **eval 用例集**（固定 prompt + 期望工具序列）在 CI 或夜间跑。

## 与本仓库（ppeng-agent-core）的对照与改进建议

| 飞轮 | 仓库内已有抓手 | 可加强方向 |
|------|----------------|------------|
| **A** | `evolution-learn` + inbox；`AGENTS.md` / Skills 沉淀偏好；Web 控制台 UX 约定 | 把「知识反馈 → 改搜集策略」**显式化**：例如在 `doc/evolution/` 增加**季度 feed 复盘**小节，或 gateway 配置旁维护「能力标签 ↔ feed 映射」表，避免只靠记忆改 RSS。 |
| **B** | `/evolution` 观测页；`doc/evolution/{success,failure,no-op}`；Daemon trace API | 缺**统一产品指标**（DAU/转化/关键路径耗时）与代码进化指标并列；可将「断链」定义为**功能矩阵**：网关 / 会话 / 沙箱 / 自愈 / 领域包逐项打勾，定期对照 no-op 理由看是**信号不足**还是**门槛过严**。 |
| **C** | Self-heal；`@ppeng/agent-sre` 只读 `k8s_get` 等；sandbox 分层 | 生产向仍缺 **K8s/Helm 一键部署与 SLI/SLO**；短期可把 **SRE persona + 外部可观测后端**（Prometheus/Loki 链接写进 domain 配置说明）写进运维文档，形成「监控 → issue → PR」模板。 |
| **D** | `npm run evolution --` 管线；已收紧的 `gateway.config.example.json` feeds | 增加**「能力维度 checklist」**（运行时 / 控制台 / Evolution / 插件）每次 learn 前扫一眼，决定本轮 `--items` 与 agent 组合；重大扩展走 **Domain 包**（`doc/DOMAIN_AGENTS.md`）而非堆 RSS。 |
| **E** | 沙箱、审批、网关、Evolution `.env` 白名单 | 安全需求清单 → CI 门禁（依赖扫描、secret 扫描、关键路径权限测试）；重大变更走 **Loop E 与 C** 双签。 |
| **F** | 模型/并发/压缩相关 env；Evolution 资源消耗 | 为 daemon / evolution 定义**预算与配额**文档；关键路径加延迟与成本字段进 Loop B 仪表盘。 |
| **G** | HTTP API、A2UI、MCP、插件清单 | API 变更与 **CHANGELOG** 联动；契约测试目录与 PR 模板勾选「破坏性变更」。 |
| **H** | 会话与 evolution 轨迹、Skills | 固定 **eval 集** + 夜间/CI 跑；将高频失败模式沉淀进 `AGENTS.md` 或 Skills。 |

**总括**：**主四环（A–D）**与**辅四环（E–H）**在真实发布中会**重叠**（例如一次 hotfix 同时走 C 与 E）。关键是把每条反馈边落成**可执行工件**（配置、门禁、仪表盘、复盘纪要）。本仓库当前最强的是 **A/D（Evolution）** 与 **C 的开发期半边（Self-heal、SRE 只读）**；**B（产品级指标）**、**F（单位经济显式化）**、**H（可重复 eval）**相对最薄，优先补齐通常性价比最高。
